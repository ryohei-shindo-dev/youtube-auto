# Incident: note予約投稿が間違った月で設定されている（カレンダー月送り未実装）

**日付**: 2026-03-29
**影響**: 4月予約の29記事全てが3月の日付で予約設定されている可能性。既に1本が意図しないタイミングで公開
**深刻度**: high

## 事象

3/29 12:30に2本のnote記事が同時公開された:
1. 「AIに投資相談するとき、先に決めておきたい『聞くこと』と『聞かないこと』」— 正規予約（3/29 12:30）
2. 「暴落で積み立てを止めるか迷ったとき」— **本来は4/10 21:00予約**

2本目は `full_schedule.json` では 4/10 21:00 に予約されているが、noteのダッシュボードでは 3/29 12:30 に公開されている。

## 直接原因

`note_publish.py` の予約投稿カレンダー操作（722行目）が**月の送り操作をしていない**。

```python
# 問題のコード
date_cell = page.wait_for_selector(
    f'.react-datepicker__day--0{day:02d}:not(.react-datepicker__day--outside-month)',
    timeout=5000,
)
date_cell.click()
```

このセレクタは `day` の番号（例: 10）だけでセルを特定する。カレンダーが3月を表示している状態で4月10日を予約しようとすると、**3月10日のセルをクリックしてしまう**。

3月10日は設定時点で過去日のため、noteが予約を即時公開扱いにしたか、または最も近い未来の日時に自動調整した可能性がある。

## 間接原因

1. 予約投稿バッチ実行時、カレンダーの表示月と予約対象月の差分を考慮していなかった
2. 予約設定後の確認（設定された日時のスクリーンショットや読み取り検証）がなかった
3. `full_schedule.json` の日付と実際にnoteに設定された日付の突合チェックがなかった

## 影響範囲

4月に予約された29記事全てが**3月の日付で予約設定されている可能性**がある。過去日になった記事からnoteが順次公開している恐れがある。

**緊急確認が必要:**
- noteのダッシュボードで全予約記事の実際の予約日時を確認
- 意図せず公開された記事がないか確認

## 暫定対応

- [x] 全29記事の予約日時を自動確認（tmp_check_schedules.py）
- [x] 結果: 正しい24本 / 不一致2本 / 公開済み3本
- [x] 不一致2本を修正（tmp_fix_2schedules_v4.py で修正・検証済み）
  - `n270ff52be0d5` 4/3 21:00 → 4/6 21:00
  - `nb2beda1db1b4` 4/1 12:30 → 4/16 12:30
- 公開済み3本はそのまま（ユーザー判断）:
  - `n524d55cb6892` 積み立て20年（予定: 4/5）
  - `naaa4579ff970` オルカンかS&P500か（予定: 4/9）
  - `n740693e9fa62` 暴落で積み立てを止めるか（予定: 4/10）

## 恒久対応

`note_publish.py` のカレンダー操作に月送りロジックを追加:

```python
# 予約月がカレンダーの表示月と異なる場合、「次月」ボタンをクリック
# 必要な回数だけ月を送る
```

加えて:
- 予約設定後に設定された日時を読み取って検証
- `full_schedule.json` の日時と突合チェック

## 再発防止策

| 間接原因 | 対策 |
|---|---|
| カレンダー月送り未実装 | 予約月と表示月の差分だけ「次月」ボタンをクリック |
| 予約設定後の検証なし | 設定後にカレンダーの表示日時を読み取って突合 |
| 過去日の予約に対するガードなし | 予約日時が現在より前の場合はエラーにする |
